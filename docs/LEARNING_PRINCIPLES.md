# Learning Principles

## Purpose

This document captures the learning principles that guide the Engineering OS. Each principle has been discovered through discussion, validated through experience, and intentionally adopted during the learning journey.

This is a living document. New principles are added only after they have been observed, discussed, and agreed upon through practice.

---

## Core Principles

| Principle | Description | Status |
|-----------|-------------|--------|
| Foundations Before Complexity | Build a strong foundation before introducing advanced topics. A well-designed foundation makes future learning faster and more sustainable. | ✅ Adopted |
| Progressive Disclosure | Introduce concepts gradually. Learn the right amount at the right time instead of trying to understand everything at once. | ✅ Adopted |
| Understand Before Memorizing | Prioritize understanding over memorization. Long-term retention naturally follows genuine comprehension. | ✅ Adopted |
| Concepts Before Tools | Learn the underlying engineering concepts before focusing on specific tools, technologies, or commands. | ✅ Adopted |
| Design Before Implementation | Think through the architecture, structure, and approach before beginning implementation. | ✅ Adopted |
| Challenge Before Acceptance | Question ideas, explore alternatives, and refine solutions instead of accepting the first proposal. Better designs emerge through constructive discussion. | ✅ Adopted |
| Single Responsibility for Documentation | Every document should answer one primary question and serve one clear purpose. | ✅ Adopted |
| No Empty Artifacts | Create files, folders, and documentation only when they provide immediate value or have a clearly defined purpose. | ✅ Adopted |
| Build for Scale | Design repositories, documentation, and systems so they remain organized and maintainable as they grow. | ✅ Adopted |
| Continuous Refinement | Treat every design as iterative. Improvements are encouraged whenever a better solution is discovered through discussion and experience. | ✅ Adopted |
| Discussion Before Documentation | Finalize ideas through discussion before documenting them. Documentation should capture understanding rather than ongoing exploration. | ✅ Adopted |
| Documentation Should Be Lightweight | Documentation should support learning, not interrupt it. Optimize for quick updates and long-term usefulness. | ✅ Adopted |
| Learn in Context | Introduce concepts when they naturally improve understanding, even if they belong to a later phase of the curriculum. | ✅ Adopted |

---

## Evolution Guidelines

To preserve the quality and purpose of this document:

- Add new principles only after they have naturally emerged during the learning journey.
- Avoid adding generic study advice or industry clichés.
- Every principle should reflect a mindset that has been intentionally adopted through discussion and practice.
- Existing principles may be be refined as our understanding evolves while preserving the reasoning behind each change.

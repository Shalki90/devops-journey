**Document Version:** 1.0
**Sprint:** 3 — Linux Fundamentals

# Purpose

This document explains how logging works in Linux, including system logs, application logs, `systemd-journald`, `rsyslog`, log files, log rotation, and troubleshooting techniques.

Understanding Linux logging is essential for system administration, production support, DevOps, cloud infrastructure, security investigations, and troubleshooting.

---

# What is Logging?

Logging is the process of recording events generated by the operating system, applications, and services.

Logs help administrators:

* Monitor system health
* Diagnose failures
* Investigate security incidents
* Audit system activity
* Debug applications
* Analyze performance

---

# Why Logging Matters

Logs provide visibility into what happened on a system.

Without logs, troubleshooting becomes largely guesswork.

Typical uses include:

* Service failures
* Authentication issues
* Network problems
* Disk errors
* Application debugging
* Security auditing

```text
Application

      ▼
```

| ---------------- | ----------------------------- |

# systemd-journald

Modern Linux systems use:

```text
systemd-journald
```

Responsibilities:

* Collect kernel logs
* Collect service logs
* Store boot logs
* Maintain structured log entries
* Integrate with `systemctl`

---

# journalctl

Primary command:

```bash
journalctl
```

Common examples:

| Command                           | Purpose                  |
| --------------------------------- | ------------------------ |
| `journalctl`                      | View all logs            |
| `journalctl -b`                   | Current boot             |
| `journalctl -b -1`                | Previous boot            |
| `journalctl -u nginx`             | Logs for a service       |
| `journalctl -f`                   | Follow logs in real time |
| `journalctl --since "1 hour ago"` | Recent logs              |
| `journalctl -p err`               | Error logs only          |

| File                | Purpose                            |
| ------------------- | ---------------------------------- |
| `/var/log/syslog`   | General system log (Debian/Ubuntu) |
| `/var/log/messages` | General system log (RHEL family)   |
| `/var/log/auth.log` | Authentication events              |
| `/var/log/secure`   | Authentication (RHEL family)       |
| `/var/log/kern.log` | Kernel messages                    |
| `/var/log/boot.log` | Boot process                       |
| `/var/log/dmesg`    | Kernel startup messages            |
| `/var/log/cron`     | Cron jobs                          |
| `/var/log/nginx/`   | Nginx logs                         |
| `/var/log/httpd/`   | Apache logs                        |

# Viewing Log Files

Useful commands:

| Command   | Purpose             |
| --------- | ------------------- |
| `cat`     | Display entire file |
| `less`    | Scroll through logs |
| `tail`    | Last lines          |
| `tail -f` | Live monitoring     |
| `head`    | First lines         |
| `grep`    | Search logs         |

Example:

```bash
tail -f /var/log/syslog
```


* Hardware detection

* Boot diagnostics

# Log Levels

| Level     | Meaning                        |
| --------- | ------------------------------ |
| Emergency | System unusable                |
| Alert     | Immediate action required      |
| Critical  | Critical condition             |
| Error     | Error occurred                 |
| Warning   | Potential issue                |
| Notice    | Normal but significant         |
| Info      | Informational                  |
| Debug     | Detailed debugging information |

# Log Rotation

Logs continuously grow over time.

Linux uses:

```text
logrotate
```

Responsibilities:

* Rotate old logs
* Compress logs
* Remove old logs
* Prevent disks from filling

---

# Log Rotation Flow

```text
Current Log
      │
Size Limit Reached
      │
Rename
      │
Compress
      │
Create New Log
```

---

# Log Rotation Configuration

Configuration locations:

| Location              | Purpose                    |
| --------------------- | -------------------------- |
| `/etc/logrotate.conf` | Global configuration       |
| `/etc/logrotate.d/`   | Application-specific rules |

# Common Troubleshooting

| Problem             | Investigation           | Resolution                            |
| ------------------- | ----------------------- | ------------------------------------- |
| Service won't start | `journalctl -u service` | Review startup errors                 |
| Login failures      | Authentication logs     | Verify credentials and permissions    |
| Boot issues         | `journalctl -b`         | Review boot sequence                  |
| Disk full           | Check `/var/log`        | Rotate or archive logs                |
| Kernel errors       | `dmesg`                 | Investigate hardware or driver issues |

---

# Best Practices

| Practice                              | Benefit                           |
| ------------------------------------- | --------------------------------- |
| Monitor logs regularly                | Detect issues early               |
| Use `journalctl` for systemd services | Centralized troubleshooting       |
| Configure log rotation                | Prevent storage exhaustion        |
| Archive important logs                | Preserve audit history            |
| Protect log files                     | Prevent unauthorized modification |

---

# Common Mistakes

| Mistake                          | Better Practice                        |
| -------------------------------- | -------------------------------------- |
| Ignoring warning messages        | Investigate before they escalate       |
| Disabling log rotation           | Can lead to full disks                 |
| Deleting active log files        | Rotate logs instead                    |
| Storing sensitive data in logs   | Mask or avoid confidential information |
| Looking only at application logs | Review system and kernel logs as well  |
